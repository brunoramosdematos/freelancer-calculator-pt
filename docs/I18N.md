# Internationalization

## Supported Locales

The application supports three complete UI catalogues:

- `en` - English
- `pt-PT` - Português (Portugal)
- `pt-BR` - Português (Brasil)

The default and fallback locale is `en`. `pt-BR` has its own complete
catalogue and does not rely on `pt-PT` fallback messages.

## Detection And Persistence

Runtime detection uses this order:

1. Valid stored user preference.
2. `navigator.languages`, in browser preference order.
3. `navigator.language`.
4. `en`.

The localStorage key remains:

```text
freelancer-calculator-pt:locale:v1
```

Locale normalization rules:

- `en`, `en-US`, `en-GB`, and other `en-*` values map to `en`.
- `pt-BR`, `pt_BR`, and case-insensitive equivalents map to `pt-BR`.
- `pt-PT` and `pt_PT` map to `pt-PT`.
- Generic `pt` maps to `pt-PT`.
- Other Portuguese regional variants, such as `pt-AO`, map to `pt-PT`.
- Unsupported languages map to `en`.

The exact `pt-BR` match happens before the generic `pt-*` rule.

## URLs And Saved Simulations

Locale is intentionally not stored in simulator URLs, hash params, route params,
Pinia tax state, or saved simulation objects. Shared calculation URLs remain
calculation-only and backward-compatible with existing parameters.

Saved simulations restore tax state only; the interface language remains the
user's current preference. Resetting the simulator clears calculation state and
URL query parameters, but does not reset the locale preference.

## Catalogue And Metadata

Messages live under `src/i18n/locales/` and use semantic domain keys. Each
locale module exports:

- `languageName`
- `metadata`
- default `messages`

`src/i18n/locales/index.ts` is the typed source of truth for registered
catalogues, language names, and metadata. `src/i18n/metadata.ts` consumes that
single source so title and description strings are not duplicated.

Runtime locale changes update:

- `<html lang>`
- document direction, fixed to `ltr`
- `document.title`
- `meta[name="description"]`

Static `index.html` metadata remains English as crawler/default fallback.
Canonical URL, Open Graph URL, routes, robots.txt, and sitemap.xml remain
unchanged.

`src/i18n/messageParity.spec.ts` verifies that all three catalogues have
identical recursive keys, no empty strings, metadata for every supported locale,
language-switcher labels, and formatter locales.

## Intl Formatting

UI locales map explicitly to native `Intl` formatting locales:

| UI locale | Intl locale |
| --------- | ----------- |
| `en`      | `en-GB`     |
| `pt-PT`   | `pt-PT`     |
| `pt-BR`   | `pt-BR`     |

Shared formatters live in `src/i18n/formatters.ts`; Vue components use
`src/composables/useLocalizedFormatters.ts`. The tax store exposes raw numbers
and does not format display strings.

Currency uses EUR in every locale with `currencyDisplay: "narrowSymbol"`:

| Value                 | `en`        | `pt-PT`      | `pt-BR`      |
| --------------------- | ----------- | ------------ | ------------ |
| `5000`, 0 decimals    | `€5,000`    | `5 000 €`    | `€ 5.000`    |
| `1234.56`, 2 decimals | `€1,234.56` | `1 234,56 €` | `€ 1.234,56` |
| `0`, 2 decimals       | `€0.00`     | `0,00 €`     | `€ 0,00`     |

Portuguese spaces may be non-breaking Unicode spaces generated by `Intl`; do
not replace them with ordinary spaces in UI code.

Plain grouped integer examples:

- `en`: `60,000`
- `pt-PT`: `60 000`
- `pt-BR`: `60.000`

Percentages preserve store semantics where `-0.2` means `-20%` and `0.25`
means `25%`.

Saved simulation dates use `Intl.DateTimeFormat` through the same locale map
with `dateStyle: "medium"` and `timeStyle: "short"`. Localized dates are never
serialized back into storage; `createdAt` remains the original ISO string.

## Input Formatting And Parsing

`FormattedNumberInput` remains integer-only. The visible grouping follows the
active locale:

- `en`: `60,000`
- `pt-PT`: `60 000`
- `pt-BR`: `60.000`

The underlying value remains a raw number. URL values stay raw, for example
`income=60000`, and saved simulations store raw parameter strings. Switching
locale reformats the visible input without calling store setters or writing to
the router.

Parsing accepts grouped or ungrouped digits and ignores currency symbols and
grouping separators. Empty or negative input keeps the existing non-positive
handling and does not introduce decimal input support.

## Glossary

Portugal Portuguese UI vocabulary:

- Income: Rendimento
- Gross income: Rendimento bruto
- Net income: Rendimento líquido
- Tax assessment: Tributação
- Joint - one income: Conjunta - um rendimento
- Save: Guardar
- Share: Partilhar
- Reset: Repor
- Saved simulation: Simulação guardada

Brazilian Portuguese UI vocabulary:

- Income: Renda
- Gross income: Renda bruta
- Net income: Renda líquida
- Tax assessment: Forma de tributação
- Joint - one income: Conjunta - uma renda
- Save: Salvar
- Share: Compartilhar
- Reset: Redefinir
- Saved simulation: Simulação salva

Legal and fiscal terms intentionally remain Portuguese-tax specific in every
locale:

- IRS
- IAS
- RNH
- NHR / RNH
- Categoria B
- recibos verdes
- Segurança Social
- Autoridade Tributária e Aduaneira
- quociente conjugal
- IVA

Do not substitute IRS with IRPF, Segurança Social with INSS, IAS with a
Brazilian equivalent, or recibos verdes with a Brazilian tax document.

Data attributes, URL parameter names, enum values, route names, storage keys,
GitHub URLs, and user-entered simulation names are not translated.

## Adding A Locale

To add a fourth locale:

1. Add the locale code to `SUPPORTED_LOCALES` in `src/i18n/locale.ts`.
2. Add an explicit `Intl` locale mapping in `src/i18n/locale.ts`.
3. Add a complete message catalogue under `src/i18n/locales/`.
4. Export `languageName`, `metadata`, and default `messages` from the locale
   module.
5. Register the locale in `src/i18n/locales/index.ts`.
6. Extend locale detection tests, formatter tests, message parity tests, and
   Cypress i18n coverage.

Do not add localized routes, URL locale parameters, saved-simulation locale
fields, or tax-store locale state without a separate product decision.
